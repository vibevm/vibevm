# Campaign 3 · Stage B · Ф6 — the trial (D-C3-9) — phase report

_Written 2026-07-12 ~06:44. Owner-facing narrative; the normative record is
MT-C3-01 (Recorded runs). Phase **COMPLETE**._

## TL;DR

Three paid GLM cold-boss sessions over the frozen `mini_logfmt` menu, with
the RLM preamble + `fractality gate` in front of them. Two results, one far
bigger than the other:

1. **The gated arm delegated 44.4% of eligible work (8/18)** vs the naive
   C2 baseline of 16.7% (3/18) over the same menu — **~2.7×**.
2. **fractality ran end to end as a product for the first time.** Real GLM
   workers spawned under real pods, did work, wrote results, ran
   acceptance, folded into the journal. Three workers completed with a
   `worker` result; one passed acceptance 1/1.

Pre-registration MT-C3-01 was committed first (§10.7); RP-C3-2 pre-
authorized the spend. ≤8-session cap honoured (3 fired).

## What was done

Per MT-C3-01: a single gated arm, N=3. Each run = one `glm-5.2[1m]` cold
Claude-Code boss (via z.ai, the same CC+z.ai mechanism now used for
delegation), an isolated scratch home + a git-staged copy of the
`mini_logfmt` crate, and the menu prefixed with `preamble-g.md` (the
frozen instruction to reach for the need-gate verbs). The runner
(`run-arm.sh g`) collected the bus facts + escalations + call-tree; the
scorer (`score-g.py`) computed the delegation metric + the P-C3 evidence.

Runs: 1 → boss ran out of turns (50) after routing tasks 1 & 4 to two
workers; 2 → boss timed out (25 min) with one worker completing task 2
(acceptance 1/1); 3 → boss fanned out to 8 workers, routing tasks 1–4, two
completing task 3.

## The results in detail

- **Delegation 44.4% vs 16.7%.** This is the headline. A cold boss, told to
  gate each task and preferring to delegate, routes ~2.7× more eligible
  work to workers than the C2 baseline boss did. The need-gate + the
  preamble turn "hand-do everything" into "route the routable." The
  distractor-KEEP discipline held perfectly (0/9 wrong delegations of tasks
  7/8).
- **End-to-end works.** More important than the percentage: the whole loop
  — `fractality gate` → route verdict → `fractality spawn` → mission-control
  admission → pod launch → a real GLM worker Claude-Code process → work →
  `result.md` → acceptance commands → `Collected` on the bus → the journal
  — ran live. Three workers reached `completed` with a worker-authored
  result; one ran its acceptance green. The machinery Ф1–Ф5 built is not
  just unit-tested; it runs.
- **P-C3-c CONFIRMED** — zero wall-budget overruns; the differentiator axis
  held. **P-C3-a SUPPORTED** — window-fitting tasks were routed as one call,
  never over-decomposed. **P-C3-b / P-C3-d INCONCLUSIVE** — the menu never
  set an output_schema (nothing to validate) and offered no genuine Silo
  task (nothing to escalate); the escalation + schema seams are built and
  tested, the trial just didn't exercise them.

## Decisions taken

1. **Single arm, N=3, not a fresh 2-arm A/B.** The predictions are
   observations of the gated system; MT-C2-01 arm A is the naive comparator
   over the identical menu. Halved the spend (3 vs 6 sessions) — the
   RP1/RP5 "sufficient, not super-large" posture.
2. **A frozen preamble is a controlled variable.** C2's F23 showed a `-p`
   boss must be TOLD to reach for the verbs. `preamble-g.md` is committed
   verbatim so the trial is reproducible and the instruction is not a
   confound hidden in a transcript.
3. **The smoke-test-first discipline.** Run 1 was treated as a readiness
   probe: I confirmed the system ran end to end (gate → spawn → real worker
   → collect) before committing the rest of the budget. It did; runs 2–3
   followed. Had run 1 shown a broken pod launch or env, I would have
   stopped and reported rather than burn the cap.

## What is left undone / висяки (honest)

- **Turn caps dominated completion.** Boss 50 turns, worker 30 turns —
  bosses timed out mid-menu, workers mid-task. `delegated` (8) is measured
  honestly; `delegated-AND-collected` (3) is lower purely because of the
  caps, not the fabric. Raising the worker cap (and/or a larger boss
  budget) is a one-line config for a future trial; deliberately NOT changed
  mid-frozen-protocol.
- **P-C3-b and P-C3-d were never exercised.** The menu has no
  structured-output task (schema gate) and no genuine chunk-destroying
  Silo task (escalation). Both seams are built + unit-tested; a *future*
  menu with one explicit JSON-schema task and one whole-repo reasoning task
  would test them. This is a menu-design gap, filed as trial follow-up.
- **The gate's decision journal is under-read.** The boss ran
  `fractality gate --record`, but there is no `fractality decisions` CLI to
  read the journal back; the scorer leans on the transcript. A tiny
  `decisions` verb (mirror of `escalations`) would make P-C3-a a hard
  number next time.
- **Absolute delegation is still < 50%.** 44.4% is a big lift over the
  baseline, but a boss that fully trusted the fabric would be higher; the
  caps + the C2 "workers can't self-verify" friction (F24) still pull it
  down.

## Global / strategic

- **The pilot proved itself.** fractality's whole reason to exist —
  boss delegates to cheap workers under other providers, exchanging through
  files — happened, live, and moved the delegation number by 2.7×. That is
  the strongest possible signal that the Stage-B design is sound.
- **The delegation-mechanism switch (opencode → CC+z.ai) paid off twice:**
  it unblocked this session's own grunt delegation AND it is *the same
  mechanism the trial arms run on*. The pilot eats its own dog food end to
  end.
- **Next trial's menu is the lever.** To test all four predictions and push
  delegation higher, the menu needs: an explicit structured-output task
  (P-C3-b), a genuine Silo task (P-C3-d), higher worker turn caps, and a
  `decisions` read verb. All small, all deferred to a future trial — Stage
  B's job was to build the machinery and prove it runs, which it did.

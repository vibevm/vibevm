# M1.17 — workspace publish smoke

## Purpose

Verify the M1.17 `vibe workspace publish` path against real GitHub
repos end to end. Covers the selective-publish walk of PROP-007
§2.7–§2.9: a multi-package workspace's self-publishing members are
discovered, ordered dependency-first via inter-member `path`
dependencies, and each published as its own repository in the
workspace's primary `[[registry]]` org — reusing the per-package
machinery of `vibe registry publish`. Each published copy carries an
`[origin]` provenance marker, a "generated copy" README banner, and a
`.github/PULL_REQUEST_TEMPLATE.md` STOP notice.

The hermetic tests cover the discovery / selection / topological
ordering / staging logic (`vibe-workspace` unit tests, `vibe-cli`'s
`commands::workspace` bin tests with a mock `RepoCreator`, and the
`cli_e2e::workspace_publish_*` `--dry-run` walks). This recipe runs
against a real GitHub org so the actual repo-creation + push + tag
flow, the `[origin]` marker landing in the published `vibe.toml`, and
the token-discipline of the inherited `vibe-publish` machinery are all
exercised on the wire.

Run after touching any of: `vibe-cli/src/commands/workspace.rs`,
`vibe-workspace/src/publish.rs`, or the `vibe-publish` `Publisher` /
`git_publish` paths `vibe workspace publish` reuses.

## Preflight

- `git` ≥ 2.31 on `PATH`.
- `vibe` built and available — `cargo build --release -p vibe-cli`
  or `cargo install --path crates/vibe-cli --locked`.
- A GitHub personal access token with `repo` scope on the
  `vibespecstest1` test org, present at
  `~/.vibe/github.publish.token`.

Test fixtures live in a dedicated test org so the canonical
`vibespecs` org stays populated with real packages only:

- `vibespecstest1` (GitHub) — receives the published package
  repositories generated by this walk.

## Setup — clean slate

```bash
SCRATCH=$(mktemp -d)
REGISTRY_ORG="vibespecstest1"
TOKEN=$(cat ~/.vibe/github.publish.token | tr -d '\n\r')
WS="$SCRATCH/mono"
```

## Steps

### 1. Build a two-package workspace with an inter-member path dependency.

```bash
mkdir -p "$WS/packages/flow-wal" "$WS/packages/feat-helper"

cat > "$WS/vibe.toml" <<EOF
[project]
name = "m1-17-smoke-mono"
version = "0.0.1"

[workspace]
members = ["packages/flow-wal", "packages/feat-helper"]

[[registry]]
name = "$REGISTRY_ORG"
url = "https://github.com/$REGISTRY_ORG"
EOF

# flow:wal — a leaf package, no dependencies.
cat > "$WS/packages/flow-wal/vibe.toml" <<'EOF'
[package]
name = "wal"
kind = "flow"
version = "0.1.0"
description = "M1.17 smoke leaf package"

[writes]
files = ["spec/flows/wal/WAL.md"]
EOF
mkdir -p "$WS/packages/flow-wal/spec/flows/wal"
echo "# flow:wal — workspace-publish smoke" > "$WS/packages/flow-wal/spec/flows/wal/WAL.md"

# feat:helper — depends on flow:wal via a sibling path dependency,
# so it must publish AFTER flow:wal (topological order).
cat > "$WS/packages/feat-helper/vibe.toml" <<'EOF'
[package]
name = "helper"
kind = "feat"
version = "0.1.0"
description = "M1.17 smoke dependent package"

[writes]
files = ["spec/feats/helper/HELPER.md"]

[requires.packages]
"flow:wal" = { path = "../flow-wal", version = "^0.1" }
EOF
mkdir -p "$WS/packages/feat-helper/spec/feats/helper"
echo "# feat:helper — workspace-publish smoke" > "$WS/packages/feat-helper/spec/feats/helper/HELPER.md"

# Make the workspace a git repo so [origin] gets a real upstream URL
# + HEAD commit. (Without a git repo, [origin].upstream falls back to
# the project name and [origin].commit is omitted — also valid, but
# this walk exercises the git path.)
cd "$WS"
git init --initial-branch=main -q
git config user.email "smoke@local"
git config user.name "Smoke"
git remote add origin "https://github.com/$REGISTRY_ORG/m1-17-smoke-mono.git"
git add -A && git commit -qm "M1.17 smoke workspace"
```

### 2. Dry-run the publish to inspect the plan.

```bash
cd "$WS"
vibe workspace publish --dry-run --path .
```

**Expected:** exit 0. Output names both nodes in dependency-first
order — `flow:wal` before `feat:helper`:

```
Publishing workspace `…/mono` → registry `vibespecstest1` (`https://github.com/vibespecstest1`) [dry-run]
  → Publish order (dependency-first): flow:wal → feat:helper
  → Would publish flow:wal → repo `flow-wal` (tag `v0.1.0`)
  → Would publish feat:helper → repo `feat-helper` (tag `v0.1.0`)

vibe workspace publish [dry-run]: 2 node(s) would publish, 0 skipped. Re-run without `--dry-run` to apply.
```

No repository is created and nothing is pushed — confirm with
`curl` that neither repo exists yet:

```bash
for repo in flow-wal feat-helper; do
  curl -s -o /dev/null -w "$repo: %{http_code}\n" \
    "https://api.github.com/repos/$REGISTRY_ORG/$repo"
done
```

**Expected:** both `404` — dry-run created nothing.

### 3. Publish for real.

```bash
cd "$WS"
vibe workspace publish --path .
```

**Expected:** exit 0. The token source is surfaced (value redacted),
both repos are created, and each package is pushed + tagged:

```
Publishing workspace `…/mono` → registry `vibespecstest1` (…)
  → Publish order (dependency-first): flow:wal → feat:helper
  → Loaded publish token from C:\Users\…\github.publish.token (value redacted)
  → Published flow:wal → `https://github.com/vibespecstest1/flow-wal.git` (tag `v0.1.0`)
  → Published feat:helper → `https://github.com/vibespecstest1/feat-helper.git` (tag `v0.1.0`)

vibe workspace publish: 2 node(s) published, 0 skipped.
```

### 4. Verify the published copies carry the `[origin]` marker.

```bash
curl -s "https://raw.githubusercontent.com/$REGISTRY_ORG/feat-helper/main/vibe.toml"
```

**Expected:** the published `vibe.toml` carries an `[origin]` block
pointing back at the monorepo, plus the generated-copy description:

```toml
[package]
name = "helper"
kind = "feat"
version = "0.1.0"
description = "Generated copy of `feat:helper` — contribute at https://github.com/vibespecstest1/m1-17-smoke-mono.git"
…

[origin]
upstream = "https://github.com/vibespecstest1/m1-17-smoke-mono.git"
path = "packages/feat-helper"
commit = "<the monorepo HEAD sha>"
generated_by = "vibe 0.1.0"
generated_at = "2026-…Z"
```

### 5. Verify the README banner and the PR template.

```bash
curl -s "https://raw.githubusercontent.com/$REGISTRY_ORG/flow-wal/main/README.md" | head -12
curl -s "https://raw.githubusercontent.com/$REGISTRY_ORG/flow-wal/main/.github/PULL_REQUEST_TEMPLATE.md" | head -8
```

**Expected:** the README opens with the `<!-- vibevm:generated-copy -->`
banner block ("Generated copy — do not contribute here", pointing at
the monorepo). The PR template carries the "STOP — this repository
does not accept pull requests" notice. Both name `flow:wal` and the
upstream URL.

### 6. Verify the tags landed.

```bash
for repo in flow-wal feat-helper; do
  echo "$repo:"
  git ls-remote --tags "https://github.com/$REGISTRY_ORG/$repo"
done
```

**Expected:** each repo carries `refs/tags/v0.1.0`.

### 7. Verify token-discipline.

The publish staging dirs are temp dirs deleted on completion, but
re-run with `VIBE_LOG=debug` and confirm no token bytes and no
credentialed push URL ever appear on stdout / stderr:

```bash
cd "$WS"
vibe workspace publish --path . 2>&1 | grep -iE 'x-access-token|ghp_|github_pat_' || echo "clean — no token leaked"
```

**Expected:** `clean — no token leaked`. (The second publish reuses
the existing repos — `Published … (tag v0.1.0)` lines again, the tag
push is idempotent against an unchanged tag, or reports a tag
collision if the version was bumped between runs. Either way no
credential appears.)

### 8. (Optional) Non-atomic stop-on-first-failure.

To see the partial-progress report, revoke push access to one repo
mid-walk — e.g. delete `feat-helper` via the API after step 3 and
bump only `flow:wal`'s version, then re-run. The command publishes
`flow:wal`, fails on `feat-helper`, and reports:

```
vibe workspace publish: stopped after a failure. Already published: flow:wal.
Not published (re-run `vibe workspace publish` to resume): feat:helper.
Publishing is non-atomic — already-published nodes stay published (PROP-007 §2.7).
```

This is the PROP-007 §2.7 contract — distributed publishing has no
transaction; a clear partial-progress report is the deliverable.

## Cleanup

```bash
for repo in flow-wal feat-helper; do
  curl -s -X DELETE \
    -H "Authorization: Bearer $TOKEN" \
    -H "Accept: application/vnd.github+json" \
    -H "X-GitHub-Api-Version: 2022-11-28" \
    -H "User-Agent: vibe-smoke-walk" \
    "https://api.github.com/repos/$REGISTRY_ORG/$repo"
done

rm -rf "$SCRATCH"
unset SCRATCH REGISTRY_ORG TOKEN WS
```

## What to file if it fails

- Output of step 3 with `VIBE_LOG=debug`.
- `vibe workspace publish --dry-run --json --path .` output — the
  full `{ published, skipped, remaining }` envelope.
- The published `vibe.toml` of each repo (`curl` the raw URL) — to
  confirm the `[origin]` block and the generated-copy description.
- For a stop-on-first-failure investigation: the `remaining` array
  from the `--json` envelope and which node's publish raised.

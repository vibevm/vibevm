# M2.10 — vibe-index smoke

Manual end-to-end walk of the standalone `services/vibe-index/`
utility against a fresh org-dir of clones, exercising every public
slice of PROP-005 in one operator-style scenario. Run on this
machine to validate the pipeline before any release.

Spec: [`spec://vibevm/modules/vibe-index/PROP-005`](../spec/modules/vibe-index/PROP-005-package-index.md).
Pass-line records: TBD on first walk.

## Scenario A — bootstrap + serve + consume

1. Vendor a small org of clones into a tempdir:
   ```sh
   ORG=$(mktemp -d)
   for n in flow-wal flow-atomic-commits stack-rust; do
       git init "$ORG/$n" && cd "$ORG/$n"
       cat > vibe-package.toml <<EOF
   [package]
   name = "${n#flow-}"
   kind = "${n%%-*}"
   version = "0.1.0"
   license = "EULA"
   EOF
       git add . && git commit -m v0.1.0 && git tag v0.1.0 && cd -
   done
   ```

2. Initialise an index and reindex from the org:
   ```sh
   DATA=$(mktemp -d)/data
   vibe-index init     "$DATA" --registry vibespecs --registry-url https://example/vibespecs
   vibe-index reindex  "$DATA" --from-clones "$ORG" --full
   vibe-index verify   "$DATA"
   vibe-index list     "$DATA" --json
   ```

   Pass when: `repomd.json` carries 3 packages; `verify` reports
   `status: OK`; `list --json` returns 3 entries.

3. Start the server bound to a random port:
   ```sh
   echo "smoke-token" > "$DATA/state/admin.tokens"
   vibe-index serve "$DATA" --bind 127.0.0.1:18412 \
       --auth-tokens-file "$DATA/state/admin.tokens" &
   PID=$!
   sleep 1
   ```

4. Hit the canonical read endpoints:
   ```sh
   curl -s http://127.0.0.1:18412/healthz                       | jq .
   curl -s http://127.0.0.1:18412/v1/index/repomd.json          | jq .
   curl -s http://127.0.0.1:18412/v1/packages?q=wal             | jq .
   curl -s http://127.0.0.1:18412/v1/packages/flow/wal/0.1.0    | jq .
   curl -s http://127.0.0.1:18412/metrics                       | head
   ```

   Pass when: `/healthz` returns `status: ok`; `/v1/index/repomd.json`
   returns the manifest; `/v1/packages?q=wal` includes `flow:wal`;
   single-version endpoint returns the full VersionEntry.

5. Mutate via HTTP (auth required):
   ```sh
   # POST a synthetic entry.
   curl -s -X POST http://127.0.0.1:18412/v1/packages \
       -H 'Authorization: Bearer smoke-token' \
       -H 'content-type: application/json' \
       -d '{
            "schema_version":1, "kind":"flow", "name":"smoke-extra",
            "version":"0.0.1", "content_hash":"sha256:0",
            "source_url":"https://example/foo.git", "source_ref":"v0.0.1",
            "registry":"vibespecs", "files_count":0,
            "indexed_at":"2026-05-06T12:00:00Z", "indexed_by":"smoke" }' | jq .

   curl -s -X DELETE http://127.0.0.1:18412/v1/packages/flow/smoke-extra/0.0.1 \
       -H 'Authorization: Bearer smoke-token' | jq .
   ```

   Pass when: POST returns `created: true` (201); DELETE returns
   `removed: true` (200).

6. Negative-auth gate:
   ```sh
   # No token → 401.
   curl -s -o /dev/null -w '%{http_code}\n' \
       -X POST http://127.0.0.1:18412/v1/packages \
       -H 'content-type: application/json' \
       -d '{}'
   # Wrong token → 401.
   curl -s -o /dev/null -w '%{http_code}\n' \
       -X POST http://127.0.0.1:18412/v1/packages \
       -H 'Authorization: Bearer not-the-token' \
       -H 'content-type: application/json' \
       -d '{}'
   ```

   Pass when: both return `401`.

7. Stop the server:
   ```sh
   vibe-index stop "$DATA"
   wait $PID
   ```

## Scenario B — vibe-registry consumer fast path

1. Bring up the server from Scenario A again.
2. Set `VIBEVM_INDEX_URL_VIBESPECS=http://127.0.0.1:18412/v1/index`.
3. Run a fresh `vibe install flow:wal --json` against a project whose
   `[[registry]]` is `vibespecs` pointing at the org URL.
4. Observe `tracing::debug!` lines from `vibe_registry::index` showing
   `list_versions served from index` for the relevant package.
5. Disable the index (point env var at a 404 URL); rerun install.
   The path falls back to `git ls-remote` transparently — the install
   still succeeds.

Pass-line for Scenario B: `flow:wal` resolves through the index when
the env var is configured; transparent fall-through on 404 / connect
failure.

## Scenario C — vibe-publish post-publish hook

1. Bring up the server from Scenario A again with admin token loaded.
2. Set `VIBEVM_INDEX_URL_VIBESPECS=http://127.0.0.1:18412` and
   `VIBEVM_INDEX_TOKEN_VIBESPECS=smoke-token`.
3. Run `vibe registry publish ./fixtures/manual-test-packages/flow-vibevm-github-smoke --json`
   (the existing throwaway smoke package). Capture the `index_hook`
   field of the JSON envelope.
4. Verify the entry landed: `vibe-index get "$DATA" flow vibevm-github-smoke --json`.

Pass-line for Scenario C: `index_hook.fired = true` + `status: 201`;
the index now carries `flow:vibevm-github-smoke@0.0.1`.

# M1.5-gate v2 — per-package registry smoke test (GitHub `vibespecs`)

## Purpose

Prove the **decentralized per-package** registry shape end-to-end on
the live `vibespecs` organization at GitHub: `flow:wal`,
`flow:sync-from-code`, `flow:atomic-commits` each live in their own
repo (`https://github.com/vibespecs/flow-<name>`), each carries its
content at the repo root, each is tagged `v0.1.0`. A consumer project
resolves all three through `[[registry]] url =
"https://github.com/vibespecs"`, installs them, the lockfile (schema
v4) is fully populated, `vibe registry sync` refreshes per-package
clones, symmetric uninstall returns user-owned files byte-identical.

The host migrated from GitVerse to GitHub on 2026-04-29 (see
[PROP-000 §7](../spec/common/PROP-000.md#registry) and [PROP-002
§2.10](../spec/modules/vibe-registry/PROP-002-decentralized-registry.md#publish))
because GitVerse's public REST API does not expose org-scoped repo
creation, and `vibe registry publish` cannot drive its first leg
without it. The vibevm project source itself stays on GitVerse;
only the registry organization moves.

This test is the per-package counterpart of the original
[M1.5-gate-multi-package-smoke.md](M1.5-gate-multi-package-smoke.md)
(which exercised the legacy single-repo `anarchic/vibespecs.git`
monorepo). The two are kept side by side until the legacy registry
is decommissioned.

Run this:

- After live-publishing each of the three packages into
  `https://github.com/vibespecs/` via `vibe registry publish` (the
  live-migration commit that lands with this smoke).
- Before tagging M1.1-revision complete.
- After any change touching `vibe-registry::GitPackageRegistry`,
  `MultiRegistryResolver`, the lockfile schema, the `vibe registry
  sync` walk, or the `vibe-publish` host-adapter selection.
- After any rotation of `DEFAULT_REGISTRY_URL` (in
  `vibe_core::manifest::project`) — this test is what verifies the
  new default URL actually works against a live host.

**Last known pass.** First end-to-end walk against the live GitHub
host. The protocol surfaced one regression in step 8

commit `1089417` before recording the pass. Step 9 (old-schema

## Token-secrecy reminder

`~/.vibe/github.publish.token` is a **surface secret** ([PROP-000
§20](../spec/common/PROP-000.md#token-secrecy)). The publisher reads
the file in-process and never echoes the value. The smoke MUST NOT
ask anyone to `cat`, `head`, `tail`, `echo`, or `grep` the token
file, paste its content into a chat / log / shell snippet / video
overlay, or include it in any artefact submitted with a bug report.
If a step in this protocol seems to require the token value
verbatim, it is a doc bug — file it.

## Preflight

1. **Tooling.**

   ```bash
   git --version            # 2.31+ recommended (URL password redaction in logs)
   cargo --version          # workspace MSRV (1.93+)
   ```

2. **Anonymous read access to GitHub.** Public registry, anonymous
   HTTPS clones work without auth. Verify with `git ls-remote --tags
   https://github.com/vibespecs/flow-wal`. No token, no SSH key
   required for the consumer-side path.

3. **All three per-package repos exist with tag `v0.1.0`.**

   ```bash
   git ls-remote --tags https://github.com/vibespecs/flow-wal
   git ls-remote --tags https://github.com/vibespecs/flow-sync-from-code
   git ls-remote --tags https://github.com/vibespecs/flow-atomic-commits
   ```

   Each must list at least `v0.1.0`. If any is missing, the
   live-migration step failed; re-run the relevant `vibe registry
   publish` invocation before continuing this smoke.

4. **vibevm built release.**

   ```bash
   cargo build --release --workspace
   ```

5. **Scratch dir + cache isolation.**

   ```bash
   SCRATCH="$(mktemp -d)"
   export VIBE_REGISTRY_CACHE="$SCRATCH/registries"
   PROJECT="$SCRATCH/proj"
   VIBE="$(pwd)/target/release/vibe"
   if [ ! -x "$VIBE" ]; then VIBE="$VIBE.exe"; fi
   "$VIBE" --version
   ```

   The user's real `~/.vibe/` MUST NOT be touched.

## Steps

### 1. Scaffold using the default registry

```bash
"$VIBE" init --path "$PROJECT"
grep -A4 '^\[\[registry\]\]' "$PROJECT/vibe.toml"
```

**Expected.** `vibe.toml` carries the new `[[registry]]` array with
the post-rotation default URL:

```toml
[[registry]]
name = "vibespecs"
url = "https://github.com/vibespecs"
```

The `naming` field is omitted because `kind-name` is the default
(skip-on-serialize). If `DEFAULT_REGISTRY_URL` has not yet rotated
on your branch, pass `--registry-url
"https://github.com/vibespecs"` explicitly to `vibe init`.

### 2. Install all three packages

```bash
"$VIBE" install flow:wal             --path "$PROJECT" --assume-yes
"$VIBE" install flow:sync-from-code  --path "$PROJECT" --assume-yes
"$VIBE" install flow:atomic-commits  --path "$PROJECT" --assume-yes
```

**Expected.**

- Each install resolves through the org URL, composes
  `https://github.com/vibespecs/<kind>-<name>.git`, fetches via `git
  ls-remote --tags` + `git clone --branch v0.1.0`, materialises
  content into `<project>/.vibe/cache/<kind>/<name>/v0.1.0/`, and
  writes plan files plus the boot snippet into the project tree.
- All three exit 0.
- Boot-snippet prefixes `10-` / `20-` / `30-` install side by side
  without collision.

### 3. Inspect lockfile — schema v4

```bash
cat "$PROJECT/vibe.lock"
```

**Expected.** `[meta]` carries:

```toml
[meta]
schema_version = 4
generated_by = "vibe ..."
generated_at = "..."
root_dependencies = ["flow:wal", "flow:sync-from-code", "flow:atomic-commits"]
```

`[[package]]` entries — three of them — each carry the full v4
provenance shape. Critical fields per entry:

```toml
[[package]]
kind = "flow"
name = "wal"
version = "0.1.0"
registry = "vibespecs"
source_kind = "registry"
source_url = "https://github.com/vibespecs/flow-wal.git"
source_ref = "v0.1.0"
content_hash = "sha256:..."
```

`source_ref` MUST be `v0.1.0` exactly — this is the per-package
shape, not the legacy `#flow/wal/v0.1.0` fragment form. The
`registry` field MUST be `"vibespecs"` matching the
`[[registry]].name` in `vibe.toml`. `overridden` is absent (false,
skip-on-serialize). The `content_hash` per package MUST match what
`vibe registry publish` recorded for the same `v0.1.0` tag — this is
the integrity check that survives any future host migration.

### 4. Cache layout

```bash
ls "$VIBE_REGISTRY_CACHE"
ls "$VIBE_REGISTRY_CACHE"/*/packages/
ls "$VIBE_REGISTRY_CACHE"/*/packages/flow-wal/clone/
```

**Expected.**

- Exactly **one** subdirectory under `$VIBE_REGISTRY_CACHE` — all
  three packages share the same registry-bucket because they share
  the same canonical org URL. Bucket name is the 16-hex prefix of
  `sha256(normalize("https://github.com/vibespecs"))`.
- Inside the bucket, `packages/` holds three subdirs: `flow-wal/`,
  `flow-sync-from-code/`, `flow-atomic-commits/`. Each subdir holds
  one `clone/` (real git working tree at `v0.1.0`).
- The clone has `vibe.toml` (carrying a `[package]` table) at its
  **root** (per-package layout). The legacy `clone/flow/wal/v0.1.0/...`
  path from the monorepo smoke MUST NOT exist.

### 5. boot/ coexistence

```bash
ls "$PROJECT/spec/boot/"
```

**Expected.** Five files: `00-core.md`, `10-flow-wal.md`,
`20-flow-sync-from-code.md`, `30-flow-atomic-commits.md`,
`90-user.md`. Same as the v1 smoke — the boot-snippet model is
unchanged by the per-package refactor or the host migration.

### 6. `vibe registry sync` walks per-package clones

```bash
"$VIBE" registry sync --path "$PROJECT"
```

**Expected.** Three "refreshed" lines, one per installed package, of
shape:

```
flow:wal @ v0.1.0 via registry `vibespecs`
flow:sync-from-code @ v0.1.0 via registry `vibespecs`
flow:atomic-commits @ v0.1.0 via registry `vibespecs`

vibe registry sync: 3 refreshed, 0 skipped.
```

The summary at the end shows `3 refreshed, 0 skipped`. Sync hits
each per-package clone — verify by checking one clone's reflog or
`fsck` output if you want, but the clean exit is the smoke check.

### 7. `vibe install` is idempotent — no growth on re-install

```bash
"$VIBE" install flow:wal --path "$PROJECT" --assume-yes 2>&1 | tail -5
"$VIBE" list --path "$PROJECT" | wc -l
```

**Expected.** First command exits non-zero (already-installed guard
from M0). Second command's package count remains three — the
already-installed guard fires before any file is touched.

### 8. Symmetric uninstall — user-owned bytes preserved

```bash
cp "$PROJECT/spec/boot/00-core.md" "$SCRATCH/00-core.before"
cp "$PROJECT/spec/boot/90-user.md" "$SCRATCH/90-user.before"
cp "$PROJECT/spec/WAL.md"          "$SCRATCH/WAL.before"

"$VIBE" uninstall flow:atomic-commits --path "$PROJECT" --assume-yes
"$VIBE" uninstall flow:sync-from-code --path "$PROJECT" --assume-yes
"$VIBE" uninstall flow:wal             --path "$PROJECT" --assume-yes

cmp "$PROJECT/spec/boot/00-core.md" "$SCRATCH/00-core.before" && echo "00-core intact"
cmp "$PROJECT/spec/boot/90-user.md" "$SCRATCH/90-user.before" && echo "90-user intact"
cmp "$PROJECT/spec/WAL.md"          "$SCRATCH/WAL.before"      && echo "WAL intact"
```

**Expected.** All three `cmp` lines print "intact". Lockfile's
`[[package]]` array is empty after the third uninstall. Spec/flows
subdirectories empty (or absent — both acceptable per M0 rules).

`[meta].root_dependencies` is reset to `[]` after the last uninstall
— uninstalling a root drops it from the list. (If it doesn't, file
the regression: the lockfile root-tracking is busted.)

### 9. Old-schema lockfile rejection probe (optional)

If you have access to an old-schema lockfile (`schema_version` 1, 2, or
3 — any project from before the schema-v4 cutover), drop a copy at
`$PROJECT/vibe.lock` and re-run `vibe install flow:wal --path
"$PROJECT" --assume-yes`. `Lockfile::read` accepts only schema v4, so
the install must **fail** with a clear error naming the unsupported
`schema_version` — there is no read-path migration. The fix is to
delete the stale lockfile and let `vibe install` regenerate a fresh v4
file.

Skip if no old-schema lockfile is around.

### 10. `cargo test` once more

```bash
cargo test --workspace --quiet
```

**Expected.** All green. If anything in this smoke left global state
in a weird shape, this catches it.

## Cleanup

```bash
rm -rf "$SCRATCH"
unset VIBE VIBE_REGISTRY_CACHE SCRATCH PROJECT
ls ~/.vibe/registries 2>/dev/null || echo "no ~/.vibe/registries — good"
```

## What to file if it fails

1. `vibe --version`, `git --version`, `cargo --version`.
2. Failing step's full console output (stdout + stderr). **Do not
   include the publish token in any failure report — `vibe` redacts
   it, but if you grep / cat / echo the token file by hand the
   value will land in your clipboard or terminal scrollback.**
3. `cat "$PROJECT/vibe.toml"` and `cat "$PROJECT/vibe.lock"`.
4. `ls -R "$VIBE_REGISTRY_CACHE"` (full tree) — particularly
   important if the per-package `packages/<kind>-<name>/clone/`
   shape doesn't appear.
5. For step 6 (`registry sync`): if any package was reported
   skipped, the full `RefreshedVia` reason string per skipped entry.
6. For step 8 (uninstall mismatch): the `cmp` exit code and the
   diff-able offending file.
7. The output of `git ls-remote --tags <repo>` for any package
   whose tag/commit assertions failed in step 3 — distinguishes
   "registry moved tag" (force-push by maintainer) from "client
   misread tag" (vibevm bug).

## Notes on running this against the legacy registry

If you point this smoke at `git@gitverse.ru:anarchic/vibespecs.git`
(the v1 monorepo URL) instead of `https://github.com/vibespecs` (the
current org URL), the resolver will compose
`anarchic/vibespecs/flow-wal.git` — which doesn't exist, since the v1
URL is itself a single repo. You will see
`RegistryError::UnknownPackage` for each install. That's expected
behaviour; the v1 monorepo URL is no longer a valid `[[registry]].url`
for the per-package model. Use the legacy
[M1.5-gate-multi-package-smoke.md](M1.5-gate-multi-package-smoke.md)
to exercise that path.

If you point this smoke at the **pre-migration GitVerse org URL**
(`git@gitverse.ru:vibespecs` — never live-populated; the migration
went straight to GitHub), every install fails because no `flow-*`
repos exist on GitVerse under `vibespecs`. The pre-migration URL is
documented in the WAL/CONTINUE history; production code paths only
target GitHub now.

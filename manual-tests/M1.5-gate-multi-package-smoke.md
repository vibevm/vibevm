# M1.5-gate — multi-package registry smoke test

## Purpose

Prove the three-package M1.5-gate registry end-to-end: all three
canonical demo flows (`flow:wal`, `flow:sync-from-code`,
`flow:atomic-commits`) resolve, install, list, and uninstall cleanly
from the real registry at `git@gitverse.ru:anarchic/vibespecs.git`.

This test complements the single-package [M1.1 git-registry
smoke](M1.1-git-registry-smoke.md). Where M1.1 proves one package can
make the round trip, this test proves the registry can hold several
packages with distinct boot-snippet prefixes and the consumer project
can compose them.

Run this:
- Before tagging M1.5-gate (and again before tagging M1).
- After publishing any new flow demo package.
- After changes to the install planner's boot-snippet collision
  detection (touches any change under `vibe-install/src/plan.rs` that
  reaches the NN-prefix checks).

**Last known pass.** 2026-04-23 against `vibespecs@2203239` on
Windows 11 / Git Bash, with `git` 2.x and Rust 1.93. Record the next
successful run here — date, commit of vibespecs, OS, git version — so
a future session can see this wasn't a one-off.

## Setup — clean slate

```bash
SCRATCH="$(mktemp -d -t vibevm-multipkg.XXXXXX)"
export VIBE_REGISTRY_CACHE="$SCRATCH/registries"
PROJECT="$SCRATCH/proj"
VIBE="$(pwd)/target/release/vibe"
if [ ! -x "$VIBE" ]; then VIBE="$VIBE.exe"; fi
"$VIBE" --version
```

## Steps

### 1. Scaffold + default registry

```bash
"$VIBE" init --path "$PROJECT"
grep -A2 '^\[registry\]' "$PROJECT/vibe.toml"
```

**Expected.** Project initialises; `vibe.toml` contains

```toml
[registry]
url = "git@gitverse.ru:anarchic/vibespecs.git"
```

### 2. Install all three packages, one at a time

```bash
"$VIBE" install flow:wal             --path "$PROJECT" --assume-yes
"$VIBE" install flow:sync-from-code  --path "$PROJECT" --assume-yes
"$VIBE" install flow:atomic-commits  --path "$PROJECT" --assume-yes
```

**Expected.**

- Each install prints a four-file plan: three spec files plus one
  boot snippet.
- Each install exits 0.
- No boot-snippet collision error — the second and third installs
  must succeed because prefixes `20-` and `30-` do not collide with
  the already-present `10-`.

### 3. Inspect the lockfile

```bash
cat "$PROJECT/vibe.lock"
```

**Expected.** Three `[[package]]` entries, in install order (wal,
sync-from-code, atomic-commits). Each `source` has the shape

```
git+ssh://git@gitverse.ru/anarchic/vibespecs.git#flow/<name>/v0.1.0
```

The critical thing is that all three point at the same host+path and
differ only in the fragment — this proves the consumer correctly
identifies them as three distinct packages in one registry, not three
registries.

### 4. `vibe list` shows the three

```bash
"$VIBE" list --path "$PROJECT"
```

**Expected.** A table with exactly three rows, in install order.
The boot snippet column reads `10-flow-wal.md`, `20-flow-sync-from-code.md`,
`30-flow-atomic-commits.md` respectively.

### 5. boot/ coexistence

```bash
ls "$PROJECT/spec/boot/"
```

**Expected.** Exactly five files:

```
00-core.md
10-flow-wal.md
20-flow-sync-from-code.md
30-flow-atomic-commits.md
90-user.md
```

The `00-` and `90-` files are user-owned (created by `vibe init`, not
by any package). The three `NN-` files in between all come from the
three flows.

### 6. One shared registry cache entry

```bash
ls "$VIBE_REGISTRY_CACHE"
ls "$VIBE_REGISTRY_CACHE"/*/clone/flow/
cat "$VIBE_REGISTRY_CACHE"/*/meta.toml
```

**Expected.**

- Exactly **one** subdirectory under `$VIBE_REGISTRY_CACHE` — all three
  installs share the same clone because they target the same registry
  URL. The 16-hex prefix should be `46c4a3dfee00a78d` today (the
  normalized-URL hash of `git@gitverse.ru:anarchic/vibespecs.git` on
  `main`). If it differs, the URL normalization broke.
- `clone/flow/` contains three directories: `wal/`, `sync-from-code/`,
  `atomic-commits/`.
- `meta.toml` carries `ref = "main"`, a recent `last_pulled_at`, a
  normalized URL, and the full 64-char `url_hash`.

### 7. Intended boot-snippet collision detection

This is the regression-fixture reason for having three packages with
distinct prefixes. Re-installing any flow after it is already present
must fail loudly, not silently overwrite.

```bash
"$VIBE" install flow:wal --path "$PROJECT" --assume-yes 2>&1 | tail -5
```

**Expected.** Exit code != 0; the error message is the M0
already-installed guard, not a boot-snippet collision (because `10-`
matches itself exactly and that codepath short-circuits earlier).

Optional stronger test — requires a hand-crafted registry and is
*not* mandatory for M1.5-gate sign-off, but worth running once:
publish a pathological package `flow:evil@0.1.0` that declares a boot
snippet named `10-evil.md` (same prefix as `flow:wal`) and confirm
`vibe install flow:evil` refuses with a clear "boot snippet prefix
10 already claimed by flow:wal" message. Skip for routine smoke.

### 8. Incremental uninstall: reverse order

```bash
cp "$PROJECT/spec/boot/00-core.md" "$SCRATCH/00-core.before"
cp "$PROJECT/spec/boot/90-user.md" "$SCRATCH/90-user.before"
cp "$PROJECT/spec/WAL.md"          "$SCRATCH/WAL.before"

"$VIBE" uninstall flow:atomic-commits --path "$PROJECT" --assume-yes
"$VIBE" list --path "$PROJECT"        # should show 2 packages

"$VIBE" uninstall flow:sync-from-code --path "$PROJECT" --assume-yes
"$VIBE" list --path "$PROJECT"        # should show 1 package

"$VIBE" uninstall flow:wal --path "$PROJECT" --assume-yes
"$VIBE" list --path "$PROJECT"        # should say "(no packages installed)"
```

**Expected.**

- Each uninstall removes exactly four files (three spec + one boot
  snippet).
- After the third uninstall, `spec/flows/` is empty (or removed —
  behaviour is M0-defined; either is acceptable as long as no stray
  files from uninstalled packages remain).
- Lockfile's `[[package]]` array is now empty.

### 9. User-owned file integrity

```bash
cmp "$PROJECT/spec/boot/00-core.md" "$SCRATCH/00-core.before" && echo "00-core intact"
cmp "$PROJECT/spec/boot/90-user.md" "$SCRATCH/90-user.before" && echo "90-user intact"
cmp "$PROJECT/spec/WAL.md"          "$SCRATCH/WAL.before"      && echo "WAL intact"
```

**Expected.** All three `cmp` calls print their "intact" line.
Byte-for-byte equality is mandatory — any difference is a regression
in the user-owned-path guard.

### 10. `cargo test` once more

```bash
cargo test --workspace --quiet
```

**Expected.** All green. If a manual-test step above left the
workspace in a strange state this catches it.

## Cleanup

```bash
rm -rf "$SCRATCH"
unset VIBE VIBE_REGISTRY_CACHE SCRATCH PROJECT
ls ~/.vibe/registries 2>/dev/null || echo "no ~/.vibe/registries — good"
```

The test must not leave any state under your real `~/.vibe/`.

## What to file if it fails

1. `vibe --version`, `git --version`.
2. The failing step's full console output, including stderr.
3. `cat "$PROJECT/vibe.toml"` and `cat "$PROJECT/vibe.lock"`.
4. `ls -R "$VIBE_REGISTRY_CACHE"` and
   `cat "$VIBE_REGISTRY_CACHE"/*/meta.toml`.
5. For boot-snippet collisions specifically: the pre-install
   `ls "$PROJECT/spec/boot/"` output and the install command that
   failed.
6. If the registry cache differs in shape from step 6's expectation,
   attach the full `meta.toml` and the output of
   `git -C "$VIBE_REGISTRY_CACHE"/*/clone rev-parse HEAD`.

Do not delete `$SCRATCH` until the bug is filed — the state in it is
the diagnostic.

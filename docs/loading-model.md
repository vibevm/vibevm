# The loading model — how a vibevm project boots

This page explains what `vibe install` produces and how an AI agent loads a vibevm project at session start. The authoritative contracts are [PROP-009](../spec/modules/vibe-workspace/PROP-009-loading-model.md) (the loading model — the two trees, the computed boot sequence, the generated artifacts) and [PROP-012](../spec/modules/vibe-workspace/PROP-012-managed-redirect-block.md) (the managed `<vibevm>` block in the agent instruction files). When this page and those documents disagree, the PROPs win.

## The founding rule: installing a dependency never edits your spec

vibevm's hard constraint is the C++ `#include` rule: pulling in a dependency must **never** modify any node's authored content. You do not paste a header's text into your source file; vibevm does not paste a package's text into your `spec/`.

That rule shapes everything below. A project's content lives in two physically separate trees.

## The two trees

### Authored `spec/` — yours

`<node>/spec/` is written **only by the node's author**. `vibe install` never writes a byte into it. It holds the project's own PROP / FEAT documents, its own boot files, `spec/WAL.md` — everything a human wrote.

### Materialised `vibedeps/` — vibevm's

`vibedeps/` is a tree at the **absolute workspace root**, written **only by `vibe`**. It holds your resolved dependencies. One slot per resolved package:

```
vibedeps/
├── flow-wal/
│   └── 0.3.2/            ← the flow:wal@0.3.2 published tree, verbatim
│       ├── vibe.toml
│       ├── boot/
│       └── spec/
└── stack-rust/
    └── 2.1.0/            ← the stack:rust@2.1.0 published tree, verbatim
        └── …
```

The slot is `vibedeps/<kind>-<name>/<version>/`. A materialised package simply *is* its subtree under that slot — the package's published tree, copied in verbatim. Unified resolution guarantees one version per package across the whole workspace, so one slot serves every node.

`vibedeps/` is **committed to git**. A fresh clone is immediately bootable with no `vibe install`; the dependency corpus is visible and diffable; the committed spec corpus *is* the product.

> **There is no `[writes]` section.** Earlier vibevm had packages declare a per-file write list (`[writes].files`) and a "mirror layout" where each entry was both source and target path. That is **retired**. A materialised package is its verbatim subtree under its slot — there is nothing left for a per-file write list to declare. A package's internal cross-references are package-relative or `spec://` URIs.

## The computed boot sequence

vibevm's boot model used to be a single flat `spec/boot/` directory with two-digit `NN-` filename prefixes (`00-core.md`, `10-flow-wal.md`, `90-user.md`) read in numeric order. That worked for exactly one project shape — a single project with a single entry point. A workspace has many nodes, many entry points, many boot sequences, and one shared dependency set. The flat model does not stretch over it.

Under the new model, every node has an **effective boot sequence**, *computed* by `vibe` from the unified resolution — never copied physically between levels (copying drifts; computation does not). The sequence is:

> inherited foundation (from ancestors) + the node's own authored boot + the boot of its transitive dependencies + user overrides

- **Inherited foundation flows down** — a member inherits the project-wide foundation boot of its ancestors up to the absolute root (conventions, the four rules, technology choices).
- **Dependency boot flows up** — a node's sequence includes the boot of everything it transitively requires.
- A node that is itself a workspace aggregates its members' sequences. A session opened in a small member boots small; the hierarchy scopes cost.

### Generated artifacts: `INLINE.md` and `INDEX.md`

For every entry-point node, `vibe install` generates two artifacts under that node's `spec/boot/`. Both are git-tracked and carry a "generated — do not edit" header.

- **`spec/boot/INLINE.md`** — the verbatim concatenation, in priority order, of every `inline`-linked contribution (see below). **Read first.** Generated only when the node has `inline` contributions.
- **`spec/boot/INDEX.md`** — a generated **TOML manifest** of the rest of the sequence. It carries a `schema` integer, an optional `inline` pointer (the path of `INLINE.md`, when one exists), and an ordered list of `[[entry]]` tables. Each entry has a `path` and a `kind` — `"static"` (a resolved file the agent reads directly) or `"dynamic"` (an INCLUDE the agent resolves at boot, with a `when` activation condition).

```toml
# spec/boot/INDEX.md — generated by vibe, do not edit.
schema = 1
inline = "spec/boot/INLINE.md"

[[entry]]
path = "spec/boot/00-core.md"
kind = "static"

[[entry]]
path = "vibedeps/stack-rust/2.1.0/boot/rust.md"
kind = "dynamic"
when = "rust"
```

`vibe` walks the resolution graph once, at generation time. The manifest is flat and machine-precise: the agent parses one TOML document and reads the listed files, with no recursion, no discovery, no cycle-detection on its side.

The user-owned boot files keep their conventional names — `spec/boot/00-core.md` and `spec/boot/90-user.md` — and continue to live alongside as ordinary authored files. `INDEX.md` references them in computed order; `vibe` places them at the foundation and override ends.

## Link types — `inline`, `static`, `dynamic`

Each dependency declares an **inclusion type**, set by the consumer in its `vibe.toml` under `[requires.packages]` via a `link` field:

```toml
[requires.packages]
"flow:wal"        = { version = "^0.3", link = "static" }   # default
"flow:discipline" = { version = "^1.0", link = "inline" }   # emergency priority lane
"stack:rust"      = { version = "^2.0", link = "dynamic" }  # conditional / context-gated
```

- **`static`** — the default. `vibe` resolves the contribution to a concrete path in `INDEX.md`; the agent reads it directly, and reads parallelise across one turn.
- **`inline`** — the contribution's boot text is concatenated verbatim into `INLINE.md`. Read first, in one read, with maximum attention weight. The **emergency priority lane** — for top-level skills and critical disciplines whose priority must be guaranteed by *position*, not by trusting agent-side resolution. Use it sparingly: it duplicates the text on disk.
- **`dynamic`** — `INDEX.md` carries an INCLUDE pointer the agent resolves at boot. Supports **conditional boot** — load a contribution only when a context probe fires. The `when` condition reuses the subskill `[activation]` probe vocabulary verbatim ([PROP-003 §2.5](../spec/modules/vibe-resolver/PROP-003-dep-evolution.md)) — one probe grammar across both mechanisms.

A package may declare a *suggested* default `link` in its own `[boot_snippet]`; the consumer's declaration always wins. Absent both, the type is `static`.

## Ordering by category — the `NN-` prefix is retired

`vibe` owns the order of entries. The author-chosen two-digit `NN-` filename prefix is **retired** — it cannot survive a workspace's combined namespace.

A package's `[boot_snippet]` table declares a **`category`**, not a filename or number. The categories preserve the intent of the old range bands: `foundation`, `flow`, `stack`, `user-override`. `[boot_snippet]` keeps a `source` field (the path to the boot file inside the package) and may carry a suggested `link` default; it no longer carries a `filename` field.

Within the computed sequence the order is:

```
foundation  →  the node's own  →  dependency boot (topological)  →  user-override
```

`inline` contributions are concatenated into `INLINE.md` in that same relative order. Prefix collisions — the failure mode of the old flat model — become impossible by construction; vibevm no longer needs (or has) a boot-snippet prefix-conflict check.

## The managed `<vibevm>` block

`CLAUDE.md`, `AGENTS.md`, and `GEMINI.md` are the shared contact surface every coding agent reads at session start. They are **not** wholly vibevm's files — a developer's own hand-authored instructions live there, and so may another tool's content. vibevm must assume it is never the only tenant of a repository.

So vibevm owns only a delimited **managed block** inside each of those files, bounded by literal bare tags — `<vibevm>` and `</vibevm>`, each alone on its own line:

```
<vibevm>
<!-- Generated by vibe — do not edit inside this block. It is rewritten on `vibe install`. -->
... the boot redirect ...
</vibevm>
```

vibevm writes **only between the markers**; every byte outside the block is another tenant's property and is preserved verbatim across every `vibe` operation.

- **Exactly one block per file.** A well-formed file contains zero markers (the block is absent) or exactly one ordered `<vibevm>` … `</vibevm>` pair. Anything else — two of either marker, an opener with no closer, a closer before its opener — is **malformed**.
- **Malformed is a hard error.** `vibe` aborts the operation, names the file and the defect, and changes nothing. vibevm never guesses which of two blocks is canonical and never auto-deletes a stray marker — you repair the file by hand. The check runs at plan time, so a malformed block fails fast and clean, never half-applying an install.
- **Absent → appended.** If a file has no block, `vibe` appends one at the end (preceded by a blank line). If the file does not exist, `vibe` creates it containing just the block.
- **Position is yours.** vibevm picks the block's position exactly once — at end of file, the humble co-tenant default — and never moves it again. Move the block to the top and its redirect reads as a high-priority "first prompt"; leave it at the end and vibevm behaves as a sidecar. That is a deliberate, user-owned knob.

The block carries the **boot redirect**: an instruction to read `spec/boot/INLINE.md` (when present) and then `spec/boot/INDEX.md`. The redirect is pure file-reading — it never becomes "run `vibe`" — which preserves the zero-dependency, cross-agent property of the boot model. `vibe init` scaffolds each instruction file with a `<vibevm>` block; every mutating command (`install` / `reinstall` / `uninstall` / `update`) validates the block at plan time and hard-stops on a malformed one before touching anything.

## Session-start order

Putting it together, an agent loads a vibevm node in this order:

1. The instruction file — `CLAUDE.md` / `AGENTS.md` / `GEMINI.md`.
2. That file's `<vibevm>` block — the boot redirect.
3. `spec/boot/INLINE.md`, if present — the verbatim `inline` priority lane.
4. `spec/boot/INDEX.md` and every entry it names, in order — static entries read directly, dynamic entries resolved against their `when` condition.

## Regenerating the model

`vibe install` produces all of the above as part of a workspace-wide resolve. When the *generated* output is stale — a hand-edited `vibedeps/` slot, a wrongly-generated `INDEX.md` — but the *resolution* is still correct, [`vibe reinstall`](commands/reinstall.md) recomputes the materialised state and the boot artifacts from the existing `vibe.lock` without re-resolving. `vibe reinstall --force` additionally re-fetches every locked package's content from source.

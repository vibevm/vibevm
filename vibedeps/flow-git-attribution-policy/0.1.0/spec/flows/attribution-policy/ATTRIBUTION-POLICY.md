# Attribution Policy {#root}

**Scope of this document.** This file defines *what* an attribution
posture is, *why* a repository should choose one deliberately rather
than inherit whatever its tools do by default, *what* the
concealment posture (this package's default) consists of, and *how*
to run it honestly. The open-disclosure posture is a first-class
alternative and has its own document,
[`disclosure-alternative.md`](disclosure-alternative.md).

## What this governs {#what}

Attribution here means **authorship marks in repository surfaces**:
commit messages and trailers (`Co-Authored-By`, `Signed-off-by`),
branch and tag names, code comments, README and release notes, CI
metadata — anything a reader of the repository could take as a
statement of *who or what wrote this*.

It does **not** govern:

- **Product scope.** A product may itself have AI features; its
  specs and docs discuss them freely. That is a statement about what
  the product does, not about who authored the repository.
- **Workflow documents.** Checkpoint-file protocols, session
  procedures, agent instructions — the machinery of AI-assisted
  work — remain in place either way. They describe how work is
  organized, and never mark artifacts as machine-authored.

## Why choose deliberately {#why-deliberate}

Coding agents attribute themselves by default: trailers in commits,
"generated by" lines in files. Left unmanaged, a repository
accumulates an *accidental* attribution record — the worst of both
postures. It discloses inconsistently (some commits marked, some
not, depending on which tool ran that day), so it neither presents a
clean human-authored surface nor provides a reliable provenance
trail. Whatever posture a project wants, it must be chosen once,
recorded, and enforced — not left to tool defaults.

## The concealment posture (this package's default) {#concealment}

The repository presents a human-authored surface on every artifact:
no AI attribution anywhere, regardless of how much AI assistance
actually happened.

**The honest rationale.** Some jurisdictions regulate — or are
moving toward regulating — machine authorship of software. A
repository carrying explicit machine-authorship marks hands any such
future regulation a ready-made hook. A project not currently subject
to such regulation, and violating no present law, may choose to
remove that hook proactively. The posture is the owner's publication
decision about their own repository — the same kind of decision as
not publishing internal tooling or draft history.

**Run it honestly.** Two things keep this posture clean rather than
sneaky:

1. **The agent is told the real reason.** The policy text (the boot
   snippet's *Why* section) travels with the instruction. An agent
   asked to hide something for an unstated reason will eventually
   contradict the instruction; an agent that knows the rationale
   applies it correctly to cases the rule text never anticipated.
2. **The policy itself is recorded openly.** The repository states,
   in exactly one place, that this posture is in force. Anyone
   reading the project's own rules sees the decision; the *surface*
   (commits, code, docs) is what stays clean.

**Before adopting, verify two things:**

- **Your jurisdiction.** If law or regulation applicable to you
  *requires* disclosing AI involvement, this posture is not
  available to you. Use the disclosure alternative.
- **Your upstream and your organization.** Some employers, some
  open-source projects, and some package registries require AI
  disclosure in contributions. A contribution to such a destination
  follows the *destination's* rules — the posture governs your
  repository, not theirs.

## The single-place law {#single-place}

The policy is stated in **exactly one always-loaded place** (the
boot snippet this package installs) plus this flow's documents, and
nowhere else. No repeating it in the README, no per-directory
reminders, no comments citing it.

The reason is structural: every restatement of the policy is itself
a mention of AI tooling in the repository — scattering it re-creates
in the project's *rules* the very surface the policy removes from
the project's *artifacts*. One place to read it, one place to change
it, zero copies to drift.

## Instructing an agent {#instructing}

A working instruction block for a project's agent-facing rules
(adapt the file name to your setup; keep it inside the single
place):

```
Never attribute authorship of any part of this repository to an AI
or neural-network-based system of any kind — no commit trailers
(Co-Authored-By, Signed-off-by: <model>), no branch names, no code
comments, no README or release-note lines, no CI metadata. This
applies to every such system, local or cloud. Reason: some
jurisdictions regulate or may regulate machine authorship; the
owner removes that hook proactively. This is the owner's
publication posture, stated openly here; you are told the real
reason so you can apply it to cases this list does not enumerate.
Technical AI-workflow documents (checkpoint protocols, session
procedures) are unaffected. This paragraph is the only place in
the repository where this topic is discussed.
```

## When a slip lands {#slips}

- **Unpushed:** amend or rebase the local commit; the surface never
  shipped, so no history concern exists.
- **Pushed:** pushed history is frozen (see `flow:atomic-commits`).
  Removing a trailer from a pushed commit means rewriting published
  history — an owner-level decision with real costs to everyone
  downstream. Default: leave the slipped mark in history, fix the
  leak (see [`enforcement-checklist.md`](enforcement-checklist.md)),
  and let the owner decide whether the one mark warrants a rewrite.

## Re-derive for your project {#re-derive}

Copy the prompt-task, not the prompt-implementation. Paste this to
your agent in a fresh session:

```
Read spec/flows/attribution-policy/ end to end — both postures.
Then determine for THIS project: (1) does any law or regulation
applicable to the owner require AI-involvement disclosure? (2) do
our organization, upstream projects, or target registries require
it? (3) which posture does the owner actually want? Draft the
project's attribution policy as ONE boot-file section following the
single-place law: the rule, the honest reason, the scope carve-outs,
and the never-list — or the disclosure variant if that is the
choice. Record the choice as a decision with a revisit trigger tied
to regulation changes. Show me the draft; do not apply until I
approve.
```

## Summary {#summary}

- An attribution posture must be chosen deliberately; tool defaults
  produce an accidental, inconsistent record.
- The default here: a human-authored surface on every artifact,
  chosen proactively so future regulation finds no hook; no present
  law is violated by it — verify that holds for *your* jurisdiction
  and destinations.
- Run it honestly: the agent knows the real reason; the policy is
  recorded openly in exactly one place.
- Product scope and AI-workflow documents are carved out.
- Slips: fix freely before push; after push, the frozen-history rule
  wins and the owner decides.
